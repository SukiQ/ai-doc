## 需求描述

### 好的需求描述

给 AI 的指令要**具体、明确、可执行**：

```bash
# 差：太模糊
> 做一个用户管理功能

# 好：具体且完整
> 实现用户管理模块：
> - GET /api/users — 用户列表，支持分页（page, pageSize）和搜索（keyword）
> - POST /api/users — 创建用户，校验邮箱唯一性，密码 bcrypt 加密
> - PUT /api/users/:id — 更新用户信息
> - DELETE /api/users/:id — 删除用户（软删除，设 deletedAt）
> 用 Prisma 操作数据库，返回 { data, error } 格式
```

### 描述公式

```
做什么 + 怎么做（技术细节） + 约束条件
```

| 要素 | 说明 | 示例 |
|------|------|------|
| 做什么 | 功能目标 | "实现标签管理 API" |
| 怎么做 | 技术方案 | "用 Express Router，数据存 PostgreSQL" |
| 约束条件 | 限制和要求 | "需要认证，分页上限 100，字段校验用 Zod" |

### 大任务拆分

复杂功能不要一次性描述，分步进行：

```
步骤 1: 设计数据库 Schema
步骤 2: 实现 API 路由
步骤 3: 前端页面和交互
步骤 4: 联调和测试
```

每步完成确认没问题，再进入下一步。

## 方案确认（Plan 模式）

复杂任务先让 AI 规划方案，确认后再执行。

### 何时使用

- 涉及 3 个以上文件
- 架构决策或技术选型
- 不确定实现路径

### 使用方式

```
> 请先规划方案，列出需要创建/修改的文件和大致改动，等我确认后再开始实现。
```

AI 会进入 Plan 模式，输出方案后等待你确认。

**审查方案时关注：**
- 文件拆分是否合理
- 是否遗漏边界情况
- 是否遵循现有代码风格

## 编码实现

### 单次指令

直接告诉 AI 要做什么：

```
> 实现用户注册接口 POST /api/auth/register
> 校验：邮箱格式、密码 ≥ 8 位、邮箱唯一性
> 成功返回 { data: { id, email, name } }
> 失败返回 { error: { code, message } }
```

### 指定上下文

让 AI 关注特定文件：

```
> 阅读 src/utils/auth.ts，了解现有的认证工具函数。
> 然后在同样的模式下，给 Comment 模型也加上权限校验。
```

### 复用现有模式

```
> 参考 src/app/api/posts/route.ts 的写法，
> 创建 src/app/api/comments/route.ts，实现评论的 CRUD。
```

### 批量操作

一次性描述多个相关任务：

```
> 创建以下组件：
> 1. TagList — 标签列表，支持搜索和分页
> 2. TagForm — 标签表单，用于新建和编辑
> 3. TagSelector — 标签选择器，用在文章编辑页
> 样式用 shadcn/ui，遵循 src/components/ 下现有组件的写法。
```

## 迭代修改

AI 生成的代码很少一次完美，需要迭代优化。

### 指出问题

```
> 这个接口有问题：分页参数没做上限校验，pageSize 传 10000 会拖垮数据库
```

### 补充需求

```
> 再加上：创建用户时自动发送欢迎邮件，邮件模板用 src/templates/welcome.html
```

### 调整方向

```
> 不用 REST 了，改成 GraphQL，用 Apollo Server。先改服务端，再改客户端。
```

## 上下文管理

长对话中，上下文窗口会被消耗。主动管理上下文：

### 压缩上下文

```bash
/compact    # 压缩当前对话，保留关键信息
```

对话超过 20 轮或感觉 AI 响应变慢时，执行一次 `/compact`。

### 恢复对话

```bash
claude -c    # 继续上次对话
claude -r    # 选择历史对话恢复
```

### 新任务开新对话

不相关的任务不要在同一个对话中进行，避免上下文混乱。

```
# 做完标签功能后，评论功能应该开新对话
claude    # 新建对话
```

## 常用技巧

| 技巧 | 操作 | 效果 |
|------|------|------|
| 先读后改 | "先阅读 xxx，再修改 yyy" | AI 理解现有模式，输出更一致 |
| 指定风格 | "遵循 src/app/api/ 下现有写法" | 保持代码风格统一 |
| 指定版本 | "用 React 19 + Next.js 15" | 减少 AI 猜测，避免用废弃 API |
| 小步推进 | 每步确认后再继续 | 及时发现问题，减少返工 |
| 错误即修 | 报错直接贴给 AI | AI 能快速定位和修复 |
| 定期压缩 | `/compact` | 避免上下文溢出 |
